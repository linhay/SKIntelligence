# SKIntelligence TUI 技术评估报告

评估方式：基于 `gemini` 对当前 `Sources/SKICLI/TUI.swift` 及相关测试文件的一次性评估输出整理  
评估日期：2026-03-06

## 1. 总体结论

当前 TUI 实现属于高质量、轻量级、自研程度较高的终端交互实现。它不依赖重量级 TUI 库，直接基于 ANSI 转义序列、`termios`、`ioctl` 和 Swift Concurrency 组织事件循环。对 UTF-8 多字节输入、中文列宽和终端恢复的处理明显高于一般“能跑就行”的 CLI 聊天页实现。

它适合作为内部开发工具和预览版能力继续推进，但如果目标是更广泛的公共发布，仍有几项明显的工程短板需要补齐，尤其是平台兼容性和窗口 resize 响应。

## 2. 优点

- CJK 支持较好：
  - 通过 `wcwidth` 和自定义 `terminalColumnWidth` 处理宽字符列宽。
  - 输入框 viewport、消息换行、裁剪都已从字符数逻辑切到列宽逻辑。
- 输入解析扎实：
  - `TUIByteParser` 支持 ASCII、方向键、删除键和 UTF-8 多字节字符。
  - 对 `Esc` 单键与转义序列之间的歧义做了超时判定。
- 并发结构清晰：
  - 使用 `actor` 作为事件队列，输入采集与主循环消费职责分离。
  - Prompt 请求通过异步任务发起，避免阻塞主渲染循环。
- 终端恢复较稳：
  - Raw Mode 开启/关闭、光标显示、备用屏切换都通过 `defer` 收束。
- 渲染策略足够轻：
  - 采用“按行差异更新”，比每次整屏重绘更节省输出。

## 3. 主要问题

### 高优先级

- 平台绑定 Darwin：
  - 当前实现大量依赖 `Darwin`、`termios`、`ioctl`、`wcwidth`。
  - 这意味着它目前实际上只支持 macOS，无法自然扩展到 Linux 环境。
- 缺失 `SIGWINCH` 处理：
  - 终端窗口变化后，界面不会立刻响应重排。
  - 现在依赖下一次事件触发后才会重新计算布局，用户感知会比较明显。

### 中优先级

- 渲染模型仍偏字符串式：
  - 当前按行 diff 已经足够轻，但仍基于字符串拼接与补空格。
  - 在高频流式输出、远程 SSH 或网络延迟场景下，仍可能出现闪烁或输出浪费。
- 状态与 UI 逻辑耦合较深：
  - Slash menu、连接状态、输入模式、消息区、导出逻辑都集中在一个 runtime 内。
  - 当前规模还能维护，再往上扩展会明显变重。
- 一些行为仍是硬编码：
  - 消息保留数量、输入读取缓冲区、菜单能力等都写死在实现里。

### 低优先级

- 没有历史滚动/翻页能力：
  - 消息多时只能看尾部，无法回看更早内容。
- 没有鼠标支持：
  - 这对现代 TUI 不是硬性要求，但会限制复杂交互体验。

## 4. 测试覆盖评估

当前已有测试：

- `TUIByteParserTests`
  - 覆盖 ASCII 字符、UTF-8 完整字符、UTF-8 分片输入。
- `TUITerminalSizingTests`
  - 覆盖 winsize、生效环境变量、默认值 fallback。

当前仍缺失的测试：

- 渲染逻辑测试：
  - `wrap`
  - `trimToColumns`
  - `sliceByColumns`
  - 含 emoji / combining marks / 混排文本的边界行为
- 状态机测试：
  - slash menu 显示/关闭
  - 输入模式切换
  - 连接失败/断开后的状态变化
- 窗口缩放行为测试：
  - 后续若引入 `SIGWINCH`，应补对应集成测试或至少端到端探针

## 5. 改进路线

### 短期

1. 增加 `SIGWINCH` 监听，在窗口尺寸变化时立即触发重绘。
2. 给列宽相关函数补更强的单元测试，重点是 emoji、组合字符和混合文本。
3. 把 runtime 的状态更新线程约束再收紧，减少未来扩展时的竞态风险。

### 中期

1. 抽出更明确的 UI 组件边界：
   - header
   - transcript
   - input
   - slash menu
2. 引入更细粒度的虚拟缓冲区或 cell-based diff，降低重绘噪音。
3. 增加 transcript 浏览模式，支持翻页或历史查看。
4. 如果要公共发布，评估 Linux 适配层。

## 6. 是否建议发布

- 作为内部开发工具或预览版：可以继续发布。
- 作为公共生产级 TUI 主入口：建议至少补齐以下两项后再发布：
  1. resize 响应
  2. 更完整的渲染边界测试

## 7. 我们的判断

这份评估和当前实际冒烟结果基本一致。现阶段 TUI 的核心链路已经能稳定工作：

- 中文输入可用
- 宽字符列宽已修
- 异常终端尺寸已有 fallback

但它仍然更像“一个工程上靠谱的专用 TUI”，还不是“成熟的跨环境终端前端框架”。如果后续把 `ski` 明确定位成终端聊天工具，短期最值得投入的就是 resize 与渲染测试，而不是继续堆菜单功能。
